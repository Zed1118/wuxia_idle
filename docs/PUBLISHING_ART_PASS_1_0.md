## 0. 一句话判断

当前版本更像“系统、数值、内容基本跑通的测试版”，不是“可以上架展示的游戏成品”。

下一阶段不应继续优先堆系统，而应开一个独立的 **1.0 出版美术阶段**，目标是让已有玩法在视觉、动效、音频和界面组织上看起来像一个完整武侠游戏。

## 0.5 工程现实锚点（必读 · 2026-05-31 Claude Code 代码实测补充）

> 本节是开工前必读的代码现实校准，**修正并优先于**正文 §3.4 / §5.1 / §5.2 / §8 的若干表述——下文写于产品视角，本节写于代码实测视角，冲突以本节为准。

### 0.5.1 当前已有的脚手架（不要重建）

代码实测发现，"出版美术"要做的很多东西**底层框架已经存在**，本阶段是"补素材 + 抽组件 + 增强动效"，不是从零搭建：

| 维度 | 现状（已存在） | 关键文件 |

| 角色头像 | `CharacterAvatar` 双层降级：iconPath→Image.asset→首字占位，errorBuilder 已具备；`BattleCharacter.iconPath` 字段已存在且已消费 | `character_avatar.dart:36-66` |
| 主菜单 | 山景背景(mountain_bg.png opacity 0.25) + 水墨 `_MenuButton`(panel+border 双文本层) | `main_menu.dart:100-113, 420-456` |

| 颜色体系 | `WuxiaColors` 已定义：gangMeng(绛红)/lingQiao(铜金)/yinRou(紫)/resultHighlight(金)/hp 三段/面板灰阶 | `lib/shared/theme/colors.dart` |
| 屏震工具 | `screen_shake.dart` 已有 | `lib/shared/effects/screen_shake.dart` |
| 窗口 | 1280×720（**无最小尺寸约束**，见 §19） | `windows/runner/main.cpp:28-32` |

→ **正文 §5.2「第一阶段做法」的"建战斗视觉"语义改为"补素材 + 抽组件 + 增强已有动效"；§5.1/§12.8「release 隐藏 debug」改为"已达成，加回归测试守护"。**

### 0.5.2 头号硬约束：战斗是"回放"不是"实时"

**战斗引擎先 `runToEnd` 把整场战斗算完，UI 再回放 `actionLog`**（`battle_strategy.dart` 纯计算 + `battle_screen.dart:310-339` listen `actionLog.length` 增长触发 `_playAction`）：

- 所有战斗视觉反馈（出招/受击/暴击/大招/胜负）**必须挂在"回放某一条 `BattleAction`"上**，不能假设"实时一拍一拍打"。
- 正文 §5.2 中"出招/受击"等措辞，工程语义一律理解为"回放对应 `BattleAction` 时播放 X"。
- 群战（`MassBattleStrategy`）是一次性跑完所有 wave 的批量结果回放，Boss/群战仪式同样走 log 回放，不是 wave 实时切换。

**这是最容易跑偏的地方**：任何描述成"实时战斗动画"的美术/代码任务，都要先翻译成"log 回放动画"。

### 0.5.3 真正的工作量在哪

- **不在写渲染逻辑**（头像/背景/动效脚手架已在），**在美术资产 + 动效打磨 + 组件抽取**。
- `assets/scenes/`（场景插画）和 `assets/techniques/`（心法图）是 **0 文件**——纯 greenfield 美术产出，成本最高。
- characters(3) / enemies(22) / maps(5) / equipment(icon~81 + detail 35) 有底子，可复用起步。
- "补头像"的真实动作 = 给角色/敌人填 `iconPath` + 出图，渲染代码已就绪。

## 0.6 方向决议（2026-05-31 · 样图审美锚定后用户拍板）

基于用户提供的 6 张参考样图（`桌面/样板图片/`，水墨修仙养成 UI 风格——**借视觉语言，不借修仙设定**）+ 4 项方向问答：

| 决议 | 选择 | 影响 |
|---|---|---|
| **光效尺度** | 水墨为主 + 关键节点适度彩光 | **微调 §3.3 红线**：日常 UI 守低饱和水墨；大招 / 突破 / 飞升等关键高光时刻允许克制彩光仪式（紫色内丹漩涡级，**非** 01 蓝电 / 05 五行高饱和彩光） |
| **战斗屏** | 本期先轻改，重点放养成 / UI 屏 | Phase A 第一刀不含战斗屏深改；战斗屏放后续轻量批 |
| **第一刀** | 主菜单（门面 + Steam 首图） | Phase A 首个垂直切片 = 主菜单（02 山门样图参考最充分、转化率最高） |
| **精度** | B 中等独立游戏级 | 关键屏（主菜单 / 突破 / 战斗）精修，次要屏够用即可 |

**视觉语言锚（从 6 样图提炼，落回武侠）**：宣纸米白面板（墨晕边）· 青灰+淡紫晕染水墨远山 / 云雾留白 · 等距水墨山门（02）· 圆形墨框头像+流派色边 · **竖排毛笔书法**（03，秘籍感）· 卷轴框+织锦边（06）· 厚涂留白立绘（04）· 分段进度条（绿）+ 金 GRADE 徽章 · 铜金 = `resultHighlight`（神物 / 收益 / 胜利）· 绛红 = `gangMeng`（危险 / 散功 / 印章）。**题材落回武侠**：不出金丹 / 五行 / Soul Formation，走学徒→武圣 / 心法 / 装备。

## 1. 为什么现在需要这个阶段

### 1.1 当前项目已完成的东西

根据现有项目文件和进度记录，当前已经具备：

- 主线、战斗、装备、心法、闭关、飞升、门派等核心系统。
- 大量配置和文案数据。
- 装备 icon 美术已批量入库。
- 部分装备 detail 图已存在。
- 角色立绘、敌人图、闭关地图、UI 水墨素材已经有基础库。
- C 段视觉验收已经多轮通过，说明“资源能显示、流程能走、关键状态不白屏”。

### 1.2 当前缺少的东西

缺的不是“有没有图片”，而是：

- 缺统一的游戏画面语言。
- 缺首屏商品感。
- 缺战斗存在感。
- 缺角色和 Boss 的压迫感。
- 缺界面层次、材质、动效、音频共同形成的体验。
- debug / Phase / seed 这类开发气质还没有和玩家版本彻底隔离。

### 1.3 现有规划为什么不够

当前 `ROADMAP_1_0.md` / `RELEASE_CHECKLIST_1_0.md` 中确实有：

- C 段视觉验收。
- H 段 UX polish。
- E 段音频。
- 美术素材批量生产记录。

但这些更偏：

- 验证 UI 没坏。
- 验证资源接上了。
- 验证按钮颜色符合水墨约束。
- 验证流程能走通。

它们还没有把“出版级游戏画面”作为正式 gate。

所以需要新增一个阶段：

> **1.0 出版美术阶段：把系统可运行版本包装成可对外展示的完整游戏。**

## 2. 阶段目标

### 2.1 总目标

玩家第一次打开游戏，在 10 秒内应该能感受到：

- 这是一个武侠游戏。
- 这是一个有审美方向的买断制独立游戏。
- 不是 Flutter 测试工具。
- 不是后台管理界面。
- 不是素材随便拼上的 Demo。

### 2.2 不追求的目标

本阶段不追求：

- 增加新玩法系统。
- 做 3D。
- 做复杂实时动作战斗。
- 做全屏手游特效。
- 做豪华 AAA 级动态演出。
- 一次性重做所有页面。

### 2.3 追求的目标

本阶段追求：

- 核心屏幕统一成一个完整美术风格。
- 战斗、角色、装备、心法有明确物象。
- Boss 战和神物掉落有仪式感。
- UI 从“功能列表”变成“江湖界面”。
- 音频有最小闭环。
- Steam 截图可以直接从游戏内截取。

## 3. 美术方向

### 3.1 核心关键词

- 暗黑厚涂水墨
- 写实武侠
- 低饱和灰墨
- 宣纸米黄
- 暗红印章
- 铜金点缀
- 山水留白
- 雾气、雨、旧纸、墨痕
- 江湖萧瑟感

### 3.2 现有素材中可继承的方向

当前可作为风格基准的素材：

- `assets/ui/mountain_bg.png`  
  适合作为主菜单、章节、过场远景基调。

- `assets/ui/paper_bg.png`  
  适合作为详情页、弹窗、文案阅读背景。

- `assets/characters/founder.png`  
  角色厚重感、写实感、江湖老派气质较强。

- `assets/maps/*.png`  
  闭关地图已经接近水墨意境方向。

- `assets/equipment/*_detail.png`  
  装备 detail 图有器物展示感，可以继续发展成装备系统视觉核心。

- `assets/enemies/*.png`  
  部分敌人 / Boss 素材可以进入战斗头像或 Boss 卡面。

### 3.3 禁止方向

不做：

- 仙侠大光效。
- 手游抽卡式角色卡。
- 大面积紫蓝渐变。
- 网游装备爆光柱。
- 过度金色。
- Q 版。
- 日漫赛璐璐。
- 材质很干净的现代 UI。
- Material 默认蓝紫按钮。
- 过饱和红、橙、蓝。

### 3.4 视觉比例

推荐画面比例：

- 70% 灰墨 / 暗色 / 宣纸底。
- 20% 低饱和青灰、冷绿、铁色。
- 5-8% 暗红印章 / 血色 / 门派色。
- 2-5% 铜金 / 神物 / 胜利收益点缀。

铜金只用于：

- 神物。
- 正向收益确认。
- 胜利。
- 章节完成。
- 高阶装备 / 高阶突破。

绛红只用于：

- 破坏性确认。
- 散功。
- 危险。
- Boss 压迫。
- 门派/江湖冲突。

## 4. 当前素材资产盘点

### 4.1 已有资产类别

当前 `assets/` 下已有：

- `assets/characters/`：3 张角色立绘。
- `assets/enemies/`：约 20+ 张敌人 / Boss 图。

- `assets/techniques/`：目录存在，但心法图尚未形成体系。

### 4.2 已有资产的价值

已有资产不是废稿。它们可以承担：

- 主菜单水墨背景。

## 5. 出版级核心屏幕

本阶段优先做 5 个屏幕。不要全项目铺开。

### 5.1 主菜单

#### 当前问题

主菜单目前更像功能入口集合。即使有山景和水墨素材，整体仍容易显得像调试面板。

#### 目标

主菜单必须成为游戏门面。

#### 画面要求

- 首屏应有完整水墨山门 / 江湖远景。
- 游戏标题应该是第一视觉中心。
- 主要入口数量压缩，避免一屏密密麻麻。
- 按钮应像“江湖卷轴 / 木牌 / 印章入口”，不是普通 Flutter 卡片。
- release 版本必须隐藏 Phase 1 / Phase 2 / seed / debug 入口。
- 入口锁态要克制：灰墨、锁印、小字原因，不做刺眼 disabled。

#### 建议入口分组

主入口：

- 继续江湖
- 主线
- 角色
- 装备
- 心法
- 闭关

次入口：

- 师徒
- 江湖
- 门派
- 排行榜
- 设置

开发入口：

- 仅 debug build 可见。
- release build 不存在。

#### 验收标准

截一张主菜单图，不解释项目背景，旁人应能判断这是武侠游戏主界面。

---

### 5.2 战斗屏

#### 当前问题

战斗是当前最影响“像不像游戏”的屏幕。现有系统逻辑很强，但视觉上容易像数值模拟器。

#### 目标

战斗屏至少要具备：

- 场景。
- 角色。
- 敌人。
- 出招反馈。
- 伤害反馈。
- 胜负仪式。

#### 第一阶段做法

不要一次做复杂动画。先做最小可行表现：

- 背景层：根据关卡 / Boss / 场景显示水墨背景。
- 左右队伍：角色头像 / 半身像 / 墨框。
- Boss：Boss 头像更大，或有独立边框。
- 出招：墨痕 sweep / 短闪 / 轻微 scale。
- 受击：红墨一点 / 伤害数字浮起。
- 暴击：铜金数字 + 一次短震。
- 大招：屏幕短暗 + 招式名题字。
- 胜利：宣纸弹窗 + 战利品。

#### 推荐优先接入点

- `character_avatar.dart`
- `battle_screen.dart`
- `BattleCharacter.iconPath`
- `assets/enemies/*.png`
- `assets/scenes/`

#### 验收标准

不看战斗日志，只看画面，玩家也应该知道：

- 哪边是我方。
- 哪边是敌方。
- 谁是 Boss。
- 谁正在攻击。
- 大招和普通攻击不同。
- 胜利和失败不同。

---

### 5.3 章节 / 剧情

#### 当前问题

剧情已经有文案，但画面仍偏阅读器。对武侠游戏来说，章节切换需要更强场景感。

#### 目标

章节页和剧情阅读形成“翻开江湖卷宗”的感觉。

#### 改造方向

- 章节卡增加封面图。
- 章首增加全宽水墨插图。
- 关键关卡可有场景背景。
- 剧情阅读保持轻点推进，但增强纸纹、留白、落款。
- 章末 Boss 胜利后有专属过场模板。

#### 第一批场景建议

不要 30 关全做。先做：

- 第 1 章开篇封面。
- 第 2 章开篇封面。
- 第 3 章开篇封面。
- Ch1 章末 Boss 场景。
- Ch3 章末 Boss 场景。
- Ch6 终章 Boss 场景。

#### 验收标准

玩家进入主线时应感觉是在翻阅江湖章节，不是在点击列表。

---

### 5.4 角色 / 装备

#### 当前问题

装备系统内容足够，但角色和装备的展示仍偏功能面板。

#### 目标

让“人”和“器物”成为游戏记忆点。

#### 改造方向

角色页：

- 角色立绘占据更高视觉权重。
- 基础属性做成武侠档案，不像表格。
- 装备槽用器物图和部位框。
- 师承、境界、心法形成一条人物成长线。

装备页：

- 装备列表保留效率，但加入阶位视觉。
- 装备详情强化大图。
- 神物、宝物、师承遗物拥有更强边框和题字。
- 装备典故用纸纹正文，不做普通文本块。

#### 素材需求

- 补齐剩余装备 detail。
- 关键神物可额外做更高质量大图。
- 师承遗物可复用装备图，但加“传承印”。

#### 验收标准

看到一件神物时，玩家应觉得“这是值得留下的东西”，而不是列表中一条高阶数据。

---

### 5.5 心法

#### 当前问题

心法本质上是武侠题材核心，但现在容易被看成倍率系统。

#### 目标

心法要有“秘籍 / 卷轴 / 修炼”的视觉感。

#### 改造方向

- 每阶心法有通用卷轴封面。
- 三流派有纹样差异。
- 主修心法拥有更强展示区。
- 凝练、散功、切换主修各自有明确情绪。

#### 第一批心法图

推荐先做 10 张：

- 7 阶通用心法 cover。
- 刚猛高阶代表 1 张。
- 灵巧高阶代表 1 张。
- 阴柔高阶代表 1 张。

#### 验收标准

### 17.1 资产路径走 yaml，不硬编码（项目红线 §5.6）

新增的所有美术资产路径**必须配置化**，不得硬编码在 widget 里（违反项目"代码不写文案/常量"红线）：

| 关卡/Boss 场景背景 | `stages.yaml`（建议加 sceneBackgroundPath） | battle_screen 已读 sceneBackgroundPath |
| 章节封面 | `data/narratives/chapters/` chapter def | 待定字段 |
| 心法卷轴 | `techniques.yaml` | 待定字段 |

唯一例外：主菜单全局背景这类单例可保留常量，但建议也收敛。

出图未到位时不破布局、不挂 widget test。`character_avatar` 和战斗背景已是此模式，新组件强制沿用。

### 17.3 组件抽取：收敛 inline，不新建色板

§7.3 提的 8 个 `Wuxia*` 组件目前**全是 inline 实现**（未封装），但底层 `WuxiaColors` + `screen_shake` 已存在。抽组件 = 把现有 inline 用法收敛进可复用 widget，**复用现有颜色常量（gangMeng / lingQiao / resultHighlight），严禁新建第二套色板**（防分叉）。

§3.4 的用色比例可直接引用**已落地的约定**作既成基准：铜金 = `resultHighlight`（神物 / 正向收益 / 胜利，§9 dialog 8 金已做）、绛红 = `gangMeng`（破坏性 / 散功 / 危险，dispel 绛红已做）。

### 17.4 素材分两档成本

- **复用增强**（characters / enemies / maps / equipment 有底子）：填 iconPath + 增量出图。
- **从零产出**（scenes/ + techniques/ 0 文件 + 章节封面 + Boss 立绘差异化）：纯 greenfield，素材计划与排期按高成本估。
