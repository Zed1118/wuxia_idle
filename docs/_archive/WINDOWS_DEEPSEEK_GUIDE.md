## 1. 你的角色与边界

### 1.1 你做什么

写中文文案。仅此而已。具体包括：
- 主线剧情（章节卷首 / 关卡短文 / Boss 战前后）
- 装备典故（预设典故 + 延续典故模板）
- 奇遇事件（标题 / 开场 / 分支选项 / 各分支后果文本）
- 心法描述（来历 / 口诀 / 招式名与描述）
- 武学领悟招式（"听雨悟剑"等独立招式的文本）
- 江湖见闻录（百科条目）

### 1.2 你绝对不做什么

❌ 不写 Dart 代码（`.dart` 文件你看都不要看）
❌ 不动 `data/ranks.yaml` `data/equipment.yaml` `data/techniques.yaml` `data/stages.yaml` `data/encounters.yaml`（这些是 Mac/Opus 的数值文件）
❌ 不修改 `GDD.md` `CLAUDE.md`
❌ 不在文案 yaml 里塞数值字段（`damage: 1500` 这种属于 Mac 端）
❌ 不自创 id（id 由 Mac 端先在数值 yaml 中定义，你按 id 写对应文案；详见 §7）

### 1.3 角色比喻

把自己想成一个**编辑兼写手**——产品经理（Mac 端）告诉你"现在需要一段第一章第三关的开场，玩家会遇到山贼，关卡 id = `stage_01_03`"，你按规范交一段 30–100 字的文案。**你不决定剧情走向、不决定数值平衡、不决定关卡设计**。

## 2. 文件分工与目录

### 2.1 你能动的目录（仅这三个根目录及其子目录）

```
data/
├── narratives/                # 你的领地
│   ├── chapters/              # 章节卷首/卷尾
│   │   └── chapter_<NN>.yaml
│   ├── stages/                # 关卡短文
│   │   └── stage_<chapter>_<idx>.yaml
│   ├── techniques/            # 心法描述（含招式文本）
│   │   ├── <technique_id>.yaml
│   │   └── insights/          # 武学领悟独立招式
│   │       └── <move_id>.yaml
│   └── codex/                 # 江湖见闻录百科
│       └── <entry_id>.md
├── lore/                      # 装备典故
│   ├── <equipment_id>.yaml    # 单件装备的预设典故
│   └── _templates/            # 延续典故模板
│       └── <template_id>.yaml
└── events/                    # 奇遇事件文本
    └── <encounter_id>.yaml
```

**id 命名约定**：全用小写汉语拼音 + 下划线，**不带音标、不带数字尾巴**（除非业务必需，如 `stage_01_03`）。
示例：`qing_feng_jian`（青锋剑）、`ti_yun_zong`（梯云纵）、`bamboo_listen_rain`（听雨悟剑）、`jiu_yang`（九阳神功）。

### 2.2 你绝对不能动的文件

- 任何 `.dart` 文件
- `data/` 根目录下任何 `.yaml`（`ranks.yaml` `equipment.yaml` 等）
- `assets/` 下任何文件
- `GDD.md` `CLAUDE.md` `pubspec.yaml`

### 2.3 文件格式

- **yaml**：装备典故、奇遇事件、心法描述、关卡短文、章节卷首
- **markdown (.md)**：仅江湖见闻录百科用 md（因为长文本带格式更舒服）
- 编码统一 **UTF-8 无 BOM**
- 缩进统一 **2 空格**
- 长文本一律用 yaml 的 `|` 块字面量（保留换行）

## 3. Demo 阶段内容总清单

| 内容类型 | 数量目标 | 单条字数 | 总字数估算 | 优先级 |
|---|---|---|---|---|
| 主线关卡短文 | 15–20 | 30–100 | 600–2,000 | 高 |
| 章节卷首 | 3 | 200–300 | 600–900 | 高 |
| 章节卷尾 | 3 | 100–200 | 300–600 | 高 |
| 装备预设典故 | 50–80 段（30–50 件装备 × 1–3 段） | 60–150 | 3,000–12,000 | 高 |
| 延续典故模板 | 5–10 个 | 30–80 | 150–800 | 中 |
| 奇遇事件 | 20–30 | 200–500（含分支） | 4,000–15,000 | 高 |
| 心法描述 | 20–30 本 | 100–200（不含招式） | 2,000–6,000 | 高 |
| 心法配套招式 | 60–150 招（每本 3–5 招） | 40–80 | 2,400–12,000 | 高 |
| 武学领悟独立招式 | 30–50 招 | 40–80 | 1,200–4,000 | 中 |
| 江湖见闻录百科 | 15–25 条 | 200–500 | 3,000–12,500 | 中 |

**总字数估算**：约 17,000 – 65,000 字。三个月内可产出。

**推荐推进顺序**（不要一上来就铺所有内容）：
1. 第一周：第一章 3 段卷首/卷尾 + 5 个关卡短文 + 5 件装备典故 + 3 个奇遇 → 让 Mac 端能跑通"上手 1 小时"的内容
2. 第二周：第一章剩余关卡 + 主线相关心法 5 本 + 配套招式
3. 第三周：第二章全套
4. 第四周：第三章全套 + 江湖见闻录核心条目（境界 / 心法 / 装备阶 / 共鸣度）
5. 第五周：奇遇剩余 + 武学领悟招式
6. 第六周起：润色 + 装备典故铺量 + 见闻录条目铺量

## 4. 写作风格总纲

### 4.1 总基调一句话

> **金庸的厚度、古龙的留白、《剑来》的克制。沉郁青衫，不画浓妆。**

### 4.2 三档风格分配

| 场景 | 风格倾向 | 占比 |
|---|---|---|
| 章节卷首、心法来历、江湖见闻录 | **金庸**：长短句交错，有门派、有典故、有历史感 | 主体 |
| 关卡短文、Boss 战前后、奇遇开场 | **古龙**：短句为主，留白，对比，意境苍凉 | 大量 |
| 招式描述、装备典故 | **金古折中**：克制写实，少形容词，多动词 | 全部 |

### 4.3 金庸气质参考特征

- 长短句交错。一段中既有"剑光如练，缠上他左肩"这样的短句，也有"那柄剑乃是当年昆仑派老掌门用九十一日打造，自打出炉便不曾入鞘"这样的长句。
- **多名词、多动词；少形容词、少副词**。
- 写人用动作和对白，不直接写情绪。
- 有门派、有传承、有时令。地名是真的（洛阳、大理、扬州），节气是真的（清明、寒露、霜降）。
- 对白讲身份场合："小兄弟"是江湖称呼，"贵客"是商家口吻，"在下"是初见自报。

### 4.4 古龙气质参考特征

- 短句为主，常常一句独立成段。
- 名词性句子可独立成段（"雪。/ 一望无际的雪。"）。
- 强烈对比（生/死、明/暗、动/静）。
- 留白：能不说就不说，让玩家自己脑补。
- 不写"心情"，只写"动作"。

### 4.5 写实武侠的"实"指什么

- **物候实**：竹叶上的水珠、夜半的更鼓、清晨的薄霜、伏天的蝉鸣。
- **器物实**：环首刀、鬼头刀、雁翎刀；竹皮缠柄、鲨鱼皮鞘；铁、钢、玄铁的差别。
- **行话实**：江湖人不说"任务"说"差事"，不说"等级"说"道行"，不说"队友"说"同道"。
- **地理实**：用真实地名（洛阳、襄阳、嘉峪关），不用"圣光城""暮光大陆"。

## 5. 通用做与不做

### 5.1 做（DO）

✅ **写动作，不写情绪**——"他握剑的手一紧"胜过"他心中愤怒"
✅ **用具体器物名**——"鬼头刀"胜过"一把刀"
✅ **用具体时令**——"霜降前后"胜过"秋天的某一天"
✅ **用真实地名**——"洛阳城外"胜过"北方某城"
✅ **长短句交错**——三短一长，或一长两短
✅ **保持字数纪律**——超出上限 20% 就要砍
✅ **遇到不会的术语先停**——问 Mac 端要规范，别瞎编术语
✅ **同一文件改完整体读一遍**——韵律比单句精彩重要

### 5.2 不做（DON'T）

❌ **网文腔**——"瞳孔猛地一缩""眼中闪过一丝复杂""仿佛过了一个世纪"
❌ **现代词汇混入**——"情绪""状态""信息""数据""选项""目标""任务""体验""感觉" → 改用"心绪/神色/讯息/路数/法子/由头/差事/见闻/觉着"
❌ **形容词堆砌**——"绝美无双的、惊世骇俗的、举世无双的剑" → 改"一柄剑"
❌ **滥用比喻**——"仿佛""宛若""犹如""恰似"一段不超过一次
❌ **滥用四字成语**——一段密集用 ≥3 个就太满了
❌ **无意义抒情**——"这一刻，时间仿佛静止了" → 删
❌ **主角光环陈述**——"凭借着过人的天赋……" → 删
❌ **直接交代规则**——"这个奇遇会让你领悟新招式" → 改"你似有所悟"
❌ **重复 GDD / CLAUDE.md 的术语外壳**——比如不要在文案里写"二流境界""寻常货"这种系统名（可以用"小有名气""寻常铁器"代替）
❌ **emoji / 颜文字 / 网络梗**——不解释
❌ **过长段落**——一段超过 4 行（约 80 字）就要分段或砍
❌ **结尾打太满**——多用句号收，少用感叹号、省略号
❌ **抢叙事主权**——你写的是世界与场景，玩家是主角；不要写"你大喜过望""你怒不可遏"这类替玩家定情绪的句子，改"你的手停了一瞬""你的呼吸轻了"

## 6. 各类内容详细规范

### 6.1 主线关卡短文 `data/narratives/stages/`

**文件名**：`stage_<章>_<关>.yaml`，例：`stage_01_03.yaml`
**对应**：`data/stages.yaml` 中同 id 的关卡（Mac 端先有 id，你才写）

**结构**：
```yaml
id: stage_01_03                # 与 stages.yaml 一致
title: 黑风岭                  # 4-6 字
opening: |                     # 30-80 字，进入战斗前
  ...
post_victory: |                # 20-50 字，战胜后
  ...
post_defeat: |                 # 可选，10-30 字，战败后（多数关卡可省）
  ...
```

**字数**：opening 30–80 字（必须），post_victory 20–50 字（必须），post_defeat 10–30 字（可选）。
**风格**：偏古龙短句，重氛围。**不要**交代世界观背景（那是江湖见闻录的事）。
**视角**：第二人称"你"，但克制使用——多写场景、少写"你"。

### 6.2 章节卷首 / 卷尾 `data/narratives/chapters/`

**文件名**：`chapter_<NN>.yaml`，例：`chapter_01.yaml`

**结构**：
```yaml
id: chapter_01
title: 学武出山                # 4-6 字
prologue: |                   # 卷首 200-300 字
  ...
epilogue: |                   # 卷尾 100-200 字
  ...
```

**风格**：偏金庸长短交错。卷首铺设氛围与时代背景，卷尾收束本章心境。
**禁忌**：不要直接复述本章具体情节（玩家刚打过），写气氛与时令。

### 6.3 装备预设典故 `data/lore/<equipment_id>.yaml`

**对应**：`data/equipment.yaml` 中同 id 的装备

**结构**：
```yaml
id: qing_feng_jian
name: 青锋剑                    # 与 equipment.yaml 一致
default_lore:
  - text: |                    # 60-150 字
      ...
  - text: |                    # 可选第二段
      ...
  - text: |                    # 可选第三段（最多 3 段）
      ...
```

**字数**：每段 60–150 字。
**段数**：1–3 段，**寻常货 / 像样货 1 段够**，**好家伙 / 利器 2 段**，**重器 / 宝物 / 神物 3 段**。
**风格**：金庸气质。重铸造来历、前任主人、传闻轶事。**禁止**写数值加成（那是 equipment.yaml 的事）。

### 6.4 延续典故模板 `data/lore/_templates/<template_id>.yaml`

装备使用过程中触发"关键事件"时由引擎拼接生成。模板含占位符。

**结构**：
```yaml
template_id: defeat_named_boss
trigger_event: defeat_named_boss
placeholders: [boss_name, location, season]
text: |
  {season}的{location}一战，此剑曾饮{boss_name}之血。
  自那以后，剑身似比往日更沉了几分。
```

**Demo 阶段需求**：5–10 个模板。建议覆盖：
1. 击败具名 Boss
2. 救助某 NPC
3. 在某地点苦练 N 次
4. 师徒传承（前任名字 + 传承时机）
5. 突破境界时随身
6. 强化达到某级数（+10 / +15 / +19）
7. 共鸣度跨阶（趁手 / 默契 / 心剑通灵）

**字数**：每条 30–80 字。**短而精**——这种文案会高频出现，太长玩家会烦。

### 6.5 奇遇事件 `data/events/<encounter_id>.yaml`

**对应**：`data/encounters.yaml` 中同 id 的奇遇（Mac 端先有 id 与触发条件，你写文本）

**结构**：
```yaml
id: bamboo_listen_rain         # 与 encounters.yaml 一致
title: 听雨悟剑                  # 4-6 字
opening: |                     # 80-200 字
  ...
choices:
  - text: <选项一文字>            # 4-10 字的动作短语
    outcome_id: insight_success # 与 encounters.yaml 中分支 id 一致
    body: |                    # 50-120 字
      ...
  - text: ...
    outcome_id: ...
    body: |
      ...
  # 通常 2-4 个分支
```

**字数**：opening 80–200，每个 body 50–120。
**分支数**：通常 2–4 个。**至少有一个"无伤而退"的选项**（拒绝 / 走开 / 静观），保护不想冒险的玩家。
**结局类型**：每个 body 结尾通常有一行隐性的果——"你似有所悟"暗示领悟、"你觉胸口一暖"暗示得物——**禁止**直接写"你获得了 XXX 招式"或"+30 修炼度"。引擎会另外显示数值反馈。

### 6.6 心法描述 `data/narratives/techniques/<technique_id>.yaml`

**对应**：`data/techniques.yaml` 中同 id 的心法

**结构**：
```yaml
id: jiu_yang
name: 九阳神功                  # 与 techniques.yaml 一致
origin: |                      # 100-200 字，心法来历
  ...
mantra: |                      # 可选，4-8 句口诀（七言或五言古风）
  ...
moves:
  - id: jiu_yang_baseline
    name: 神功护体              # 招式名 3-6 字（4 字最佳）
    description: |             # 40-80 字
      ...
  - id: jiu_yang_burst
    name: 一气贯顶
    description: |
      ...
```

**字数**：origin 100–200；mantra 可选；每招 description 40–80。
**招式数**：每本心法 3–7 招（按 心法阶 / 流派 / Mac 端 schema 决定）。

### 6.7 武学领悟独立招式 `data/narratives/techniques/insights/<move_id>.yaml`

通过奇遇/触发条件领悟的独立招式，不属于某本完整心法。

**结构**：
```yaml
id: ting_yu_jian
name: 听雨
description: |                 # 40-80 字
  雨势既密，剑势愈轻。出手时身形不动，剑尖已至。
  此招看似无力，实则将周身雨势尽化为剑意。
prerequisite_hint: |           # 30-60 字，"如何领悟"的氛围提示（不直接说条件）
  竹林中的雨夜，若你已斩落足够多的剑客……
```

**注意**：`prerequisite_hint` 只是氛围暗示，**真正的触发条件在 Mac 端的 `encounters.yaml`**，你不要在这里写"需要击败 100 个剑客"这种数据化的话。

### 6.8 江湖见闻录百科 `data/narratives/codex/<entry_id>.md`

百科条目，玩家在游戏内"江湖见闻录"菜单中查看。

**结构**：直接 markdown，不需要 yaml 头。

```markdown
# 境界

武学一道，自古以来便有"九品"之分。然江湖人言粗犷，
不问九品而论"流"——三流、二流、一流，再上去就是
绝顶、宗师、武圣。每一阶又分七层……

（此处写 200-500 字）
```

**字数**：200–500。
**风格**：百科腔，可有"据某某老镖头说"、"江湖中人皆知"、"近百年来罕见"等口语化转引，**不要**写成纯系统说明。

**Demo 必备条目**（建议）：
- 境界与道行
- 心法与流派
- 装备与品阶
- 共鸣度（人剑合一）
- 强化与磨剑石
- 闭关与时辰
- 师徒传承
- 三大派 / 主要门派概况
- 江湖通用规矩（拜帖、镖局、押礼、放话）

## 7. 与 Mac 端的协作约定

### 7.1 id 联结规则（铁律）

你写的每一个文案文件都通过 `id` 字段与 Mac 端的数值文件挂钩。**id 由 Mac 端先定义**，你按 id 写对应文案。

**id 不一致会让游戏加载时直接抛错**——这是设计如此，避免静默错位。

### 7.2 工作流（每次产新文案）

1. **从 Mac 端拿到清单**：本批要写哪些 id？建议 Mac 端先 commit 一份"待文案清单" issue 或 markdown，列出 id + 一句话需求（如"`stage_01_03` 第一章第三关，山贼劫道，玩家此时是三流·初学"）。
2. **按 id 创建文件**：路径与命名严格按 §2.1。
3. **写完本批后跑校验**：
   ```bash
   # Mac 端会提供这个脚本，由你在 Windows 端运行
   dart run tool/validate_content.dart
   ```
   这个脚本会检查 id 是否齐全、字段是否缺、字数是否超限。**校验红了不要 commit**。
4. **commit 前再读一遍**：连续读 3 段以上的话，听一下自己写的"调子"是不是统一。

### 7.3 遇到不确定怎么办

| 情况 | 处理 |
|---|---|
| 不知道某 id 对应什么 | 不写，问 Mac 端 |
| 觉得 Mac 端给的关卡设计有问题 | 不擅自改剧情，开 issue 讨论 |
| 想新增一个奇遇 id | 不行——id 必须 Mac 端先建（含触发条件与数值），你才写文案 |
| 字数怎么也压不下来 | 砍内容，不放宽字数 |
| 不会写某个题材 | 在该 id 下放占位 `TODO: 待补`，commit 时标 `[draft]` 前缀 |

### 7.4 不要做的协作动作

❌ 不要去改 `data/encounters.yaml` 加一个 id 然后再去 events 写——id 必须 Mac 端先建
❌ 不要在 events yaml 里塞 `damage`、`reward` 等数值字段
❌ 不要在文案里写出具体数字（"扣 50 内力"），改"内力大损"这种描述性表达
❌ 不要因为字数超了就分多个 yaml 文件——一个 id 一个文件，超字数砍内容

## 8. 高质量完整样本

### 8.1 样本一：主线剧情（关卡短文）

**`data/narratives/stages/stage_01_03.yaml`**

```yaml
id: stage_01_03
title: 黑风岭
opening: |
  山道转过一道弯，风声忽止。
  三十余条汉子从两侧的乱石后站起身来。
  当中一个独眼大汉提了把鬼头刀，咧嘴笑道：
  "这位小哥，山高路远，留下些买路钱再走。"

post_victory: |
  独眼大汉跪在地上，手指扣进碎石。
  你看着他腰间那把环首刀，刀鞘上錾着春水堂的印记——
  半年前，洛阳到许昌那一趟失镖的物件。
```

**为什么这是好样本**：
- opening 70 字，post_victory 50 字，符合规范
- 古龙气质：短句、白描、留白
- 实物：鬼头刀、环首刀、春水堂、洛阳、许昌（地名都是真的）
- 行话：买路钱、失镖、镖局印记
- 后文埋了一个钩子（春水堂、失镖）——以后剧情可能呼应
- 没有写"你大喜过望"或"你愤怒地"——把情绪留给玩家

### 8.2 样本二：装备典故

**`data/lore/qing_feng_jian.yaml`**

```yaml
id: qing_feng_jian
name: 青锋剑
default_lore:
  - text: |
      此剑出自洛阳「春水堂」，是十年前万掌柜亲手所铸。
      剑身青莹，入水不沉，斩铁不卷。
      万掌柜当年放话：此剑当配三流以上的人，方不算糟蹋了它。

  - text: |
      剑柄缠的是雨后采来的青竹皮，握在手里凉而不滑。
      据说万掌柜的小徒弟曾用此剑赢过一场比武——
      挑落了对手帽上的红缨，却没伤分毫。
      此事在洛阳茶馆里传了大半年。
```

**为什么这是好样本**：
- 两段共 130 字，符合"好家伙"阶 2 段规范
- 金庸气质：有铸造来历、有人物（万掌柜及其徒弟）、有地名（洛阳）、有传闻
- 物候细节：雨后采的青竹皮，凉而不滑
- 不写数值：没出现"+10 攻击力"
- 留白：万掌柜为人如何？小徒弟后来如何？玩家自己想

### 8.3 样本三：奇遇事件（完整带分支）

**`data/events/bamboo_listen_rain.yaml`**

```yaml
id: bamboo_listen_rain
title: 听雨悟剑
opening: |
  竹叶上水珠成串而下，雨声渐密。
  你立在林间，剑入鞘已半个时辰。
  忽然之间，你想起这百日来斩落的剑影——
  原来每一招的轻重缓急，都与这雨势暗合。
  你按剑的手停住了。

choices:
  - text: 闭目静听
    outcome_id: insight_success
    body: |
      雨声更重，剑意更轻。
      再睁眼时，林中已无雨。
      你提剑划出一道弧，竹叶纷落，无一片着身。

  - text: 拔剑试招
    outcome_id: practice_partial
    body: |
      你拔剑出鞘，依雨势使了一遍剑法。
      招式仍是旧招，却觉手腕灵活了几分。
      雨停时，你还在原地，剑尖朝下。

  - text: 转身离去
    outcome_id: skip
    body: |
      此情此景，未必再有。
      但雨终归是要停的，路终归是要走的。
      你提了剑，下山去了。
```

**为什么这是好样本**：
- opening 95 字，每个 body 50-60 字，全文约 280 字，符合规范
- 三个分支覆盖三种玩家心态：钻研型 / 实操型 / 求稳型
- "转身离去"是无伤而退选项——玩家不想冒险也有出路
- 暗示性结局：
  - "竹叶纷落，无一片着身"暗示领悟新招（不直说）
  - "手腕灵活了几分"暗示修炼度增加（不直说）
  - "下山去了"暗示无收获但无损失（不直说）
- 真实物候：竹林、雨声、半个时辰
- 古龙气质：短句、留白、不直接说情绪

## 9. 提交规范

### 9.1 commit message

- 中文，动宾，简明
- 文案类一律加 `[content]` 前缀
- 草稿状态加 `[draft]` 前缀
- 涉及多类型（同时写了关卡 + 奇遇）的 commit 在描述中分行说明

**示例**：
```
[content] 写第一章 5 个关卡短文与 3 个奇遇

- stages: stage_01_01 / 01_02 / 01_03 / 01_04 / 01_05
- events: bamboo_listen_rain / old_well_echo / wandering_monk
```

### 9.2 commit 节奏

- **不要堆几十个文件一次性 commit**——按"一批 5–10 个文件"为单位
- **校验通过才 commit**（见 §7.2）
- **每天结束前推一次远端**——避免 Windows 机器出问题丢稿

### 9.3 PR 描述模板

```
## 本次内容
- 类型：关卡短文 / 装备典故 / 奇遇 / 心法 / 见闻录
- id 列表：xxx, yyy, zzz
- 总字数：约 NNN 字

## 自检清单
- [ ] id 与 Mac 端数值 yaml 一致
- [ ] 字数全部在规范区间内
- [ ] 跑过 validate_content.dart 通过
- [ ] 通读一遍韵律统一
- [ ] 没有出现 §5.2 的禁忌词

## 待 Mac 端确认
（如果有任何不确定的设计点，写在这里）
```
